How to set up, configure, and run Professor Tux end to end

Professor Tux is a FastAPI app with a student chat UI at /, an admin panel at /admin, uploaded lecture retrieval, and an Ollama-compatible model backend. This page covers local setup, Ollama local and cloud setup, admin configuration, lecture uploads, teaching modes, student usage, storage paths, and troubleshooting.

Overview

Professor Tux has four main layers:

  1. Student UI at / for learner-facing chat.
  2. Admin UI at /admin for runtime control, uploads, and logs.
  3. FastAPI backend for sessions, chat, modes, uploads, and health.
  4. Ollama-compatible model backend for generation, with local Ollama as the default.
Important behavior This app uses an Ollama-compatible /api/chat endpoint. That means you can run it with local Ollama, a remote Ollama-compatible host, or local Ollama forwarding to Ollama Cloud.

Requirements

  • Python 3.10+
  • pip
  • Ollama installed on the machine running Professor Tux, or access to another Ollama-compatible host
  • Enough disk and RAM for your chosen model and lecture embedding index
Before exposing this app to other users Change the default admin credentials. Out of the box they are admin / professortux.

Quick Start

This is the fastest reliable way to get Professor Tux running locally.

1. Clone the repository 
git clone https://github.com/hvbhanot/ProfessorTux.git
cd ProfessorTux
2. Create a virtual environment 
python3 -m venv .venv
source .venv/bin/activate

On Windows PowerShell:

python -m venv .venv
.venv\Scripts\Activate.ps1
3. Install Python dependencies 
pip install -r requirements.txt
4. Copy the example environment file 
cp .env.example .env
5. Start Ollama and make sure a model exists 
ollama serve
ollama pull qwen3.5:4b
6. Run Professor Tux 
python run.py
7. Open the app
  • Student UI: http://localhost:8000/
  • Admin UI: http://localhost:8000/admin
  • API docs: http://localhost:8000/api/docs
  • Product docs: http://localhost:8000/docs
Repository Source code lives at github.com/hvbhanot/ProfessorTux.

Day-1 Operator Checklist

If this is your first time setting up Professor Tux, use this sequence. It covers the actual order that avoids the most common misconfiguration issues.

1. Confirm the backend is live

Open http://localhost:8000/health or load / in the browser. If the server is not reachable, fix that before doing any admin work.

2. Sign in to /admin and verify the model backend

Check the Ollama URL first. Then enter a model name and click Run. Do not move on until the admin page shows the model as ready.

3. Set the default student mode intentionally

Use Guided Learning if this is a teaching assistant. Use Recall only if the student needs tiny cues during an exam and should not receive direct answers.

4. Upload course material if you want course-grounded teaching

Upload lecture PDFs, decks, or notes. Wait for ingestion to finish, then confirm the lecture list and chunk count increase in admin.

5. Validate Guided mode with a course-specific question

Ask something that clearly depends on the uploaded content. If Guided mode still answers generically, check lecture context, ingestion status, and the uploaded files.

6. Validate Recall mode with a direct-answer request

Ask for a hint-only response such as Give me one small hint about XSS, do not answer directly. Recall should respond with a nudge, not a solution.

Ollama Local Setup

Local Ollama is the simplest and most reliable path.

Recommended local flow

ollama serve
ollama pull qwen3.5:4b

Then set these values in .env if needed:

OLLAMA_BASE_URL=http://127.0.0.1:11434
OLLAMA_MODEL=qwen3.5:4b
OLLAMA_KEEP_ALIVE=5m

Ollama Cloud Setup

For chat, Professor Tux talks to an Ollama-compatible local endpoint. The local daemon handles cloud model forwarding after you sign in. For CTF live web search, Professor Tux calls Ollama's authenticated web-search API directly with a saved admin API key or OLLAMA_API_KEY.

1. Start your local Ollama daemon 
ollama serve
2. Sign in on the same host where Professor Tux is running 
ollama signin
3. Keep the local base URL pointed at your local Ollama 
OLLAMA_BASE_URL=http://127.0.0.1:11434
4. In the admin panel, run a cloud model name 
gpt-oss:120b-cloud
5. Optional: enable CTF live web search

Create an Ollama API key, then paste it into Admin → Runtime → CTF Web Search. You can also add it to .env and restart Professor Tux.

OLLAMA_API_KEY=your_ollama_api_key
OLLAMA_WEB_SEARCH_BASE_URL=https://ollama.com
Common misunderstanding Do not point OLLAMA_BASE_URL at an ollama.com web URL. It should still be the local or remote Ollama-compatible daemon endpoint that exposes /api/chat. Web search uses OLLAMA_WEB_SEARCH_BASE_URL separately.

Admin Setup

The admin page is where you finish runtime configuration after the server is up.

Login

Open /admin and sign in with the configured admin credentials. Defaults are:

username: admin
password: professortux

Recommended first-time admin flow

  1. Verify the Ollama API URL is correct.
  2. Enter a model name and click Run.
  3. Pick the student default mode.
  4. Set an optional default topic.
  5. Set an optional course filter.
  6. Decide whether lecture slide context should be enabled by default.
  7. Upload lecture materials if you want course-grounded Guided Learning responses.

Student Usage

The student page is intentionally simple. The learner does not manage runtime state there; they mainly choose a mode and ask questions.

Guided Learning mode

Guided Learning is for teaching, explanation, worked walkthroughs, and structured help. It is also the mode that uses uploaded lecture material when lecture context is enabled.

  • Use this when the student wants explanation, examples, or step-by-step teaching.
  • Uploaded lecture material can be pulled in and cited.
  • Cloud and local models both use the same backend path.
Explain SQL injection from first principles.
Walk me through the incident response lifecycle step by step.
Use the uploaded lecture notes to explain ARP spoofing in simple terms.

Recall mode

Recall mode is intentionally constrained for in-exam use.

  • It gives one small hint or cue instead of a full answer.
  • It is intentionally stricter for MCQs and option-based questions.
  • It does not currently inject uploaded lecture material.
Give me one hint about Kerberos tickets.
Nudge me toward the answer for this MCQ without telling me the option.
One quick cue only: what should I remember about CSRF?
Mode expectation If the student wants notes, tables, worked examples, or full answers, they should be in Guided Learning. Recall mode is intentionally the opposite: minimal cues under pressure.

Sessions

Student chat sessions are created in memory. If the backend restarts, those sessions are lost and the browser may need to recreate the session.

What the student can and cannot control

  • The student can ask questions, continue the current session, and use the currently active learning mode.
  • The student cannot switch model backends, upload lectures, or change ingestion settings from the student page.
  • Admin defaults affect new sessions; they do not rewrite old in-memory sessions automatically.

Lectures & RAG

Professor Tux supports lecture-grounded answering through a ChromaDB-backed retrieval pipeline.

Supported upload types

  • .pdf
  • .pptx
  • .ppt
  • .txt
  • .md

What happens on upload

  1. The file is copied to UPLOAD_DIR.
  2. Text is extracted.
  3. Text is chunked with overlap.
  4. Chunks are embedded with the configured embedding model.
  5. Embeddings and metadata are stored in ChromaDB.

How uploaded material is used

  • Guided mode now prefetches relevant lecture context before generation.
  • Guided mode can still use the lecture search tool after that if needed.
  • Recall mode intentionally skips lecture retrieval to stay minimal.
If uploads seem ignored Check that lecture context is enabled in admin, that documents were actually ingested, and that you are testing in Guided mode rather than Recall mode.

Teaching Modes

Modes live in app/modes/ and are auto-discovered at startup.

app/modes/
├── guided.md
├── recall.md
└── README.md

Creating a new mode

  1. Create a new .md file under app/modes/.
  2. Add YAML frontmatter with id, name, and optional metadata.
  3. Write the system prompt body below the frontmatter.
  4. Reload modes from admin or restart the server.

For the mode file structure, see app/modes/README.md.

Deployment & Security

The current codebase is optimized for local or controlled-network usage. If you plan to expose it more broadly, tighten a few things first.

  • Change the default admin credentials in .env.
  • Put the app behind TLS using a reverse proxy if it will leave localhost.
  • Persist the data/ directory if you want uploads, embeddings, logs, and runtime settings to survive container or VM replacement.
  • Be aware that student sessions and admin auth tokens are in memory only.
  • Review CORS settings in app/main.py before exposing the app to untrusted origins.
Good deployment split Run Professor Tux and Ollama on the same machine or same trusted network segment when possible. That keeps the backend path simple and avoids cross-host auth confusion with cloud-backed Ollama sessions.

Troubleshooting

1. The student page says no model is ready

  • Open /admin.
  • Check the Ollama API URL.
  • Run a model name in the model field.
  • If local Ollama is used, make sure ollama serve is running.

2. Uploaded lectures do not affect answers

  • Make sure the files actually show up in the lecture table.
  • Make sure lecture context is enabled in admin.
  • Test in Guided Learning, not Recall mode.
  • Use prompts related to the uploaded course content.

3. Recall mode refuses to answer directly

That is expected. Recall mode is intentionally constrained to hints and cues.

4. Session not found after a restart

Sessions are in memory. Restarting the backend clears them. The browser should recreate them, but old session state is not durable.

5. Cloud model output loses spaces or formatting

If you saw words being glued together when switching to cloud models, that was caused by whitespace handling in streamed chunks. That bug has already been fixed in the current code.

6. Uploads work locally but seem ignored in cloud mode

Guided mode now prefetches lecture context before generation, so uploads should still work in cloud-backed generation. If they do not, check that you are in Guided mode, lecture context is enabled, and the uploaded content matches the question.

7. Changing admin settings does not survive restart

Only some runtime settings are persisted right now. The Ollama URL, CTF web-search settings, and max tokens are saved; other admin defaults are still runtime-only unless you add persistence for them.
