How to set up, configure, and run Professor Tux end to end

Overview

Professor Tux has four main layers:

  1. Student UI at / for learner-facing chat.
  2. Admin UI at /admin for runtime control, uploads, and logs.
  3. FastAPI backend for sessions, chat, modes, uploads, and health.
  4. Ollama-compatible model backend for generation, with local Ollama as the default.
Important behavior This app uses an Ollama-compatible /api/chat endpoint. That means you can run it with local Ollama, a remote Ollama-compatible host, or local Ollama forwarding to Ollama Cloud.

Requirements

  • Python 3.10+
  • pip
  • Ollama installed on the machine running Professor Tux, or access to another Ollama-compatible host
  • Enough disk and RAM for your chosen model and lecture embedding index
Admin credentials reset on every server restart Bootstrap login uses admin / professortux (or whatever you set in ADMIN_USERNAME / ADMIN_PASSWORD). Immediately on first sign-in, the admin panel forces you to set a fresh username and password for this server lifetime. Credentials live in memory only and are wiped when the FastAPI process restarts. See Admin Setup for the full flow.

Quick Start

This is the fastest reliable way to get Professor Tux running locally.

1. Clone the repository 
git clone https://github.com/hvbhanot/ProfessorTux.git
cd ProfessorTux
2. Create a virtual environment 
python3 -m venv .venv
source .venv/bin/activate

On Windows PowerShell:

python -m venv .venv
.venv\Scripts\Activate.ps1
3. Install Python dependencies 
pip install -r requirements.txt
4. Copy the example environment file 
cp .env.example .env
5. Start Ollama and make sure a model exists 
ollama serve
ollama pull qwen3.5:4b
6. Run Professor Tux 
python run.py
7. Open the app
  • Student UI: http://localhost:8000/
  • Admin UI: http://localhost:8000/admin
  • API docs: http://localhost:8000/api/docs
  • Product docs: http://localhost:8000/docs
Repository Source code lives at github.com/hvbhanot/ProfessorTux.

Day-1 Operator Checklist

If this is your first time setting up Professor Tux, use this sequence. It covers the actual order that avoids the most common misconfiguration issues.

sequenceDiagram autonumber participant Op as Operator participant API as FastAPI participant Adm as Admin UI participant Oll as Ollama participant Ch as ChromaDB Op->>API: GET /health API-->>Op: 200 OK Op->>Adm: Sign in to /admin (bootstrap creds) Adm-->>Op: Forced credential reset Op->>Adm: Set new username + password Op->>Adm: Set Ollama URL + model Adm->>Oll: pull / load model Oll-->>Adm: ready Op->>Adm: Pick default mode + topic Op->>Adm: Upload lecture material Adm->>Ch: chunk + embed + persist Op->>API: Test Guided question API->>Ch: retrieve top-K API->>Oll: generate Oll-->>API: streamed answer API-->>Op: cited Guided response

The order matters: confirm the backend, lock the model, set defaults, then ingest lectures before validating with a real question.

1. Confirm the backend is live

Open http://localhost:8000/health or load / in the browser. If the server is not reachable, fix that before doing any admin work.

2. Sign in to /admin and complete the forced credential reset

Sign in with the bootstrap credentials (admin / professortux by default, or your ADMIN_USERNAME / ADMIN_PASSWORD overrides). The admin panel will immediately force you to set a new username and password for this server lifetime — every other control is locked until you do.

3. Verify the model backend

Check the Ollama URL. Then enter a model name and click Run. Do not move on until the admin page shows the model as ready.

4. Set the default student mode intentionally

Use Guided Learning if this is a teaching assistant. Use Recall only if the student needs tiny cues during an exam and should not receive direct answers. CTF is for authorized offensive practice; Custom sends no system prompt and should only be used when guardrails live inside the model itself.

5. Upload course material if you want course-grounded teaching

Upload lecture PDFs, decks, or notes. Wait for ingestion to finish, then confirm the lecture list and chunk count increase in admin. Existing uploads from prior runs are still on disk; you only need to upload what's new.

6. Validate Guided mode with a course-specific question

Ask something that clearly depends on the uploaded content. If Guided mode still answers generically, check lecture context, ingestion status, and the uploaded files.

7. Validate Recall mode with a direct-answer request

Ask for a hint-only response such as Give me one small hint about XSS, do not answer directly. Recall should respond with a nudge, not a solution.

Ollama Cloud Setup

For chat, Professor Tux talks to an Ollama-compatible local endpoint. The local daemon handles cloud model forwarding after you sign in. For CTF live web search, Professor Tux calls Ollama's authenticated web-search API directly with a saved admin API key or OLLAMA_API_KEY.

Chat traffic always goes through the local daemon. CTF web search is a separate authenticated call that bypasses it.

1. Start your local Ollama daemon 
ollama serve
2. Sign in on the same host where Professor Tux is running 
ollama signin
3. Keep the local base URL pointed at your local Ollama 
OLLAMA_BASE_URL=http://127.0.0.1:11434
4. In the admin panel, run a cloud model name 
gpt-oss:120b-cloud
5. Optional: enable CTF live web search

Create an Ollama API key, then paste it into Admin → Runtime → CTF Web Search. You can also add it to .env and restart Professor Tux.

When configured, CTF Mode preloads web search results for every non-social CTF turn and can still run narrower follow-up searches when needed.

OLLAMA_API_KEY=your_ollama_api_key
OLLAMA_WEB_SEARCH_BASE_URL=https://ollama.com
Common misunderstanding Do not point OLLAMA_BASE_URL at an ollama.com web URL. It should still be the local or remote Ollama-compatible daemon endpoint that exposes /api/chat. Web search uses OLLAMA_WEB_SEARCH_BASE_URL separately.

Admin Setup

The admin page is where you finish runtime configuration after the server is up. Every server restart begins with a forced credential reset, then the rest of the runtime configuration is loaded from ./data/admin_runtime.json.

Bootstrap login

Open /admin and sign in with the bootstrap credentials. The out-of-the-box defaults are:

username: admin
password: professortux

If you would prefer different bootstrap values, set ADMIN_USERNAME and ADMIN_PASSWORD in .env before starting the server. Either way, those values are only accepted for the one bootstrap login per server lifetime — they cannot become the running credentials.

Forced credential reset

Credentials reset on every server start — this is intentional. The admin panel refuses to load any other control until you replace both the username and the password. Credentials are kept in memory only; neither the username nor the PBKDF2-SHA256 password hash is written to ./data/admin_runtime.json. When the FastAPI process restarts, both are wiped and the forced flow fires again. Plan real credentials ahead of time and expect to set them on each boot.

You can also rotate the running credentials mid-session via the Change Credentials button in the topbar; doing so signs every other admin session out.

Per-restart credential lifecycle

The bootstrap login is one-shot. The first thing the operator does on every boot is rotate to fresh credentials.

Username rules

  • 3–64 characters.
  • Letters, digits, dot, underscore, and hyphen only.
  • Cannot equal the default admin.

Password rules

  • At least 8 characters.
  • Cannot equal the default professortux.
  • Must differ from the bootstrap password you just used.

Recommended per-boot admin flow

  1. Sign in with the bootstrap credentials and immediately set fresh ones — this is non-skippable.
  2. Verify the Ollama API URL is correct (already loaded from admin_runtime.json).
  3. Enter a model name and click Run.
  4. Pick the student default mode.
  5. Set an optional default topic.
  6. Set an optional course filter.
  7. Decide whether lecture slide context should be enabled by default.
  8. Upload new lecture materials if needed (existing uploads survive the restart).

Student Usage

The student page is intentionally simple. The learner does not manage runtime state there; they mainly choose a mode and ask questions.

How modes route a question

Each mode shapes prompt assembly and tool access differently, but every turn is appended to the chat log.

Guided Learning mode

Guided Learning is for teaching, explanation, worked walkthroughs, and structured help. It is also the mode that uses uploaded lecture material when lecture context is enabled.

  • Use this when the student wants explanation, examples, or step-by-step teaching.
  • Uploaded lecture material can be pulled in and cited.
  • Cloud and local models both use the same backend path.
Explain SQL injection from first principles.
Walk me through the incident response lifecycle step by step.
Use the uploaded lecture notes to explain ARP spoofing in simple terms.

Recall mode

Recall mode is intentionally constrained for in-exam use.

  • It gives one small hint or cue instead of a full answer.
  • It is intentionally stricter for MCQs and option-based questions.
  • It does not currently inject uploaded lecture material.
Give me one hint about Kerberos tickets.
Nudge me toward the answer for this MCQ without telling me the option.
One quick cue only: what should I remember about CSRF?
Mode expectation If the student wants notes, tables, worked examples, or full answers, they should be in Guided Learning. Recall mode is intentionally the opposite: minimal cues under pressure.

CTF mode

CTF mode treats the session as authorized offensive security work — the persona's "no exploit code" rule is lifted so the model can produce concrete payloads, enumeration steps, and tool invocations.

  • Use this only for CTF challenges, wargames, or labs the student owns or is authorized against.
  • If an Ollama API key is configured, every non-social CTF turn is preloaded with web-search results for current CVEs, writeups, and tool docs.
  • Tool-capable cloud models (e.g. gpt-oss:120b-cloud) are recommended; live web search and the ctf_agent_command tool both require tool calling.

Custom mode

Custom mode sends no system prompt at all — no Professor Tux persona, no identity guard, no safety rules, no teaching style, no few-shot examples. The model receives only the topic / lecture context (if admin set them), the conversation history, and the student's message.

  • Use this to evaluate the raw behaviour of an active model, or when the model already carries its own teaching persona.
  • Whatever guardrails you need (refusing offensive use, hiding the underlying model name, scope to cybersecurity, output style) must be built into the model itself — fine-tuning, an Ollama Modelfile SYSTEM directive, or a model that already enforces them internally.
  • The admin panel shows a confirmation modal before activating Custom mode so the operator acknowledges this trade-off.
Custom mode is not a teaching mode Without the base persona, the model can leak its underlying name, ignore the cybersecurity scope, and refuse or comply with offensive requests according to its own training. Treat Custom mode as a developer / evaluation surface, not a default for students.

Sessions

Student chat sessions are created in memory. If the backend restarts, those sessions are lost and the browser may need to recreate the session.

Chat logging

Every chat turn placed through the student interface is logged. Each entry is appended to ./data/logs/chat_logs.jsonl and includes timestamp, client IP, session ID, mode, topic, the student's question, the system prompt, the model response, retrieved sources, response duration, and the model name.

  • Logs persist across restarts — they live on disk under CHAT_LOG_DIR.
  • The student UI does not surface a "private mode"; everything sent there is recorded.
  • If you deploy this to learners, disclose the logging in your acceptable-use notice.
Privacy note for operators Chat logs include raw question and response text plus the requesting IP. Treat ./data/logs/ as sensitive and keep it off shared backups unless that is intentional.

What the student can and cannot control

  • The student can ask questions, continue the current session, and use the currently active learning mode.
  • The student cannot switch model backends, upload lectures, or change ingestion settings from the student page.
  • Admin defaults affect new sessions; they do not rewrite old in-memory sessions automatically.

Lectures & RAG

Professor Tux supports lecture-grounded answering through a ChromaDB-backed retrieval pipeline.

Supported upload types

  • .pdf
  • .pptx
  • .ppt
  • .txt
  • .md

What happens on upload

  1. The file is copied to UPLOAD_DIR.
  2. Text is extracted.
  3. Text is chunked with overlap.
  4. Chunks are embedded with the configured embedding model.
  5. Embeddings and metadata are stored in ChromaDB.

How uploaded material is used

  • Guided mode now prefetches relevant lecture context before generation.
  • Guided mode can still use the lecture search tool after that if needed.
  • Recall mode intentionally skips lecture retrieval to stay minimal.
If uploads seem ignored Check that lecture context is enabled in admin, that documents were actually ingested, and that you are testing in Guided mode rather than Recall mode.

Teaching Modes

A mode is a single Markdown file in app/modes/. Drop a new .md file there, restart the server (or call POST /modes/reload from the admin panel), and the mode appears in the admin and in GET /modes.

Loader: app/mode_loader.py. Prompt assembly: ProfessorTux.build_messages in app/professor.py.

Do not delete the six built-in modes recall.md, recall_wrong.md, guided.md, guided_wrong.md, ctf.md, and custom.md have ids wired into special-case logic in app/professor.py (recall hard rules, CTF override, wrong-turn rules, custom-mode persona bypass) and into the admin confirmation modals. You can freely add new mode files alongside them, and freely delete or edit any modes you yourself add — but removing the originals will leave dead references in the codebase.

The frontmatter parser is intentionally minimal — one key: value per line, quotes stripped, no nested YAML or multi-line strings.

What the body should contain

For every non-custom mode, the body is appended to the base persona as "You are in {name}. {body}". Keep it short and behavioural:

  • A one-line statement of what this mode does.
  • A bulleted list of rules (be concise, use tables when comparing, stay on cybersecurity, etc.).
  • Optional ## Examples block with **Student:** … / **Response:** … pairs separated by ---. These become few-shot turns prepended to the conversation — useful for small models that imitate better than they instruction-follow.
  • Optional ## Suggestions block (parsed and stripped from the system prompt) for the starter cards on the student empty state. One bullet per card: - **Card title:** Prompt text sent when clicked.
500-character rule cap _split_mode_prompt joins all rule lines with single spaces and truncates at ~500 chars before injection. Long rule lists get cut off mid-sentence. Put your most important rules first, keep each bullet under ~100 chars, and restate long rules tersely. ## Examples are parsed separately and not subject to the cap.

Reserved IDs

These IDs trigger extra logic in app/professor.py on top of your body. If your mode's id matches one of these patterns, you inherit the extra rules — pick a fresh slug if you don't want them.

ID patternExtra behaviour
recall, recall_*Adds RECALL_MODE_HARD_RULES. If the student message looks like an MCQ, also adds RECALL_MODE_MCQ_RULES. Caps max_tokens at 96.
*_wrongWhen the wrong-turn flag is set on a request, adds WRONG_TURN_HARD_RULES plus the recall- or guided-flavour wrong-turn rules. Bumps temperature to ≥1.05.
ctfAdds CTF_MODE_HARD_RULES (lifts the "no exploit code" rule, enables web_search preloading + ctf_agent_command). Admin shows a confirmation modal before activating.
customSkips the base persona, the mode marker line, all overrides, and few-shot examples. Only topic / lecture context (if set) and the conversation history reach the model. Admin shows a confirmation modal.

Student-page wiring

Everything student-facing for a mode lives in this .md file — the student page reads it from the /modes API. Specifically:

  • The mode's color drives the entire student-page palette via applyModeAccent() in index.html — accent, surface, border, and text vars are derived from the hex via CSS color-mix(). No per-mode CSS class needed.
  • The four student_* frontmatter fields drive the mode pill, empty-state title, empty-state copy, composer placeholder, and brand subtitle. Generic fallbacks kick in for any field you leave blank.
  • The ## Suggestions body block drives the starter cards.
Removing a mode is a single delete Deleting a .md file removes a mode cleanly with no leftover code in index.html. A brand-new .md file shows up fully styled with no JS or CSS edits.

Reloading

  • GET /modes (public) re-scans app/modes/ on every call, so dropping a new .md file is picked up the moment the admin page refreshes — no manual reload required.
  • POST /modes/reload (admin auth) is still available as an explicit force-refresh and is what the ↻ Reload Modes button calls.
  • A full server restart also picks up changes.

Deployment & Security

The current codebase is optimized for local or controlled-network usage. If you plan to expose it more broadly, tighten a few things first.

  • Set non-default ADMIN_USERNAME / ADMIN_PASSWORD in .env for the bootstrap login. The forced credential reset still fires on every restart, so the env-var values are only used once per server lifetime.
  • Plan an out-of-band channel for the per-restart admin credentials (password manager, ops runbook, etc.) — they do not persist on disk.
  • Put the app behind TLS using a reverse proxy if it will leave localhost.
  • Persist the data/ directory if you want uploads, embeddings, logs, and runtime settings to survive container or VM replacement. Admin credentials are deliberately excluded from data/.
  • Be aware that student sessions, admin auth tokens, and admin credentials are all in memory only.
  • Review CORS settings in app/main.py before exposing the app to untrusted origins.
Good deployment split Run Professor Tux and Ollama on the same machine or same trusted network segment when possible. That keeps the backend path simple and avoids cross-host auth confusion with cloud-backed Ollama sessions.

Tune Ollama for concurrency

Set these on the Jetson before starting the daemon. They are server-side, not client-side — they belong on the box running ollama serve, not the box running Professor Tux.

OLLAMA_NUM_PARALLEL=4
OLLAMA_MAX_LOADED_MODELS=1
OLLAMA_KEEP_ALIVE=30m
OLLAMA_FLASH_ATTENTION=1
OLLAMA_HOST=0.0.0.0:11434
  • OLLAMA_NUM_PARALLEL=4 — runs four concurrent generations on the same loaded model. Remaining students queue.
  • OLLAMA_MAX_LOADED_MODELS=1 — prevents the daemon from juggling models in and out of the 8 GB shared memory.
  • OLLAMA_KEEP_ALIVE=30m — keeps the model resident between bursts so the first request after a quiet period does not pay the load cost.
  • OLLAMA_FLASH_ATTENTION=1 — reduces KV cache memory and speeds up long contexts on Ampere hardware.

Reality check

Twenty truly-simultaneous, long-context generations on one Jetson are not realistic regardless of model size — the GPU is the bottleneck. In a real class of 20, only 3–6 students are typing at any instant, which is what the parallel-slots configuration is sized for. If lectures are bursty (everyone hits Send at once during a lab), expect 5–15 second waits before tokens start streaming. Streaming responses are already on by default, which hides the queue.

Recommended starting point Run llama3.2:3b-instruct-q4_K_M or qwen2.5:3b-instruct-q4_K_M on an Orin Nano 8GB with OLLAMA_NUM_PARALLEL=4. Watch tegrastats and nvidia-smi during a class. If the GPU saturates and queues grow, drop to a 1.5B model. If answer quality is the limit, move to an Orin NX 16GB or split the load across two Jetsons behind a load balancer.

Web server on a different host

When Professor Tux runs separately from Ollama:

  • On the Jetson, expose Ollama with OLLAMA_HOST=0.0.0.0:11434 and confirm a firewall rule allows the web-server host to reach it.
  • On the Professor Tux host, set OLLAMA_BASE_URL=http://<jetson-ip>:11434 in .env (or in Admin → Ollama API URL).
  • Keep both machines on the same trusted LAN. Ollama has no built-in auth, so do not expose the Jetson's 11434 port to the public internet.

Reducing latency in the Nano + thin-client split

When the model lives on the Jetson and Professor Tux runs on another box (a laptop, mini-PC, or "thin client"), perceived latency comes from three places: the Nano's tokens-per-second, the network hop between the two machines, and the prompt size. Tune them in this order — earlier items have far larger impact than later ones.

1. Confirm the Nano is using its GPU

This is by far the biggest variable. If layers are not offloaded, you are running on the Nano's CPU and tokens-per-second drops 5–10×. While a chat is generating, on the Nano run:

nvidia-smi --query-gpu=memory.used,utilization.gpu --format=csv -l 1
  • If utilization.gpu stays under ~50%, the model is CPU-bound — pick a smaller quantization or smaller model.
  • If memory.used sits at the cap, the model is swapping — drop a tier (e.g. 3B → 1.5B, or Q4 → Q3).
  • Check Ollama's startup log for offloaded N/N layers to GPU. Anything less than full offload means the rest is on CPU.

2. Pick a quantized model that fits

See the model table above. On an Orin Nano 8GB the sweet spot is a 3B Q4 model (llama3.2:3b-instruct-q4_K_M or qwen2.5:3b-instruct-q4_K_M). On the original 4 GB Nano, drop to 1.5B Q4 — anything larger will swap. Quantization is the single biggest knob for tokens-per-second.

3. Keep the model warm

The repo default OLLAMA_KEEP_ALIVE=5m unloads the model after five idle minutes, so the next first-token after a quiet stretch pays the full reload cost (5–15 s on a Nano). Bump it on the Jetson:

OLLAMA_KEEP_ALIVE=24h

4. Pin the network path

RTT between the thin client and the Nano should be 5–30 ms on a wired LAN. If you measure 100+ ms, latency is being added by something:

  • Use wired ethernet on both machines, ideally the same switch. Wi-Fi adds 10–40 ms of jitter.
  • Use a static IP for the Jetson and put it in OLLAMA_BASE_URL directly. mDNS resolution can stall.
  • Avoid Tailscale / Wireguard / VPN tunnels for this hop — they add 20–50 ms per RTT and the streaming responses amplify that.

5. Shrink the prompt where it makes sense

Smaller prompt = smaller KV cache = faster first token and faster steady-state. The biggest wins come from RAG settings, since lecture context can balloon prompt length:

  • Drop TOP_K_RESULTS from 5 to 3 in .env on the thin client.
  • Drop CHUNK_SIZE from 500 to 350 (and re-upload lectures so they re-chunk).
  • Disable lecture context in admin for modes where it is not needed. Recall mode already skips RAG; Custom mode also sends a minimal prompt.

6. Stream, do not block

The web UI already calls POST /chat/stream, so the user sees tokens as they generate even when total time is slow. Only the blocking POST /chat endpoint waits for the full response — useful for tests, not for users.

7. Keep FastAPI on the thin client, not the Nano

Embedding the lectures with all-MiniLM-L6-v2 is CPU-friendly and runs fine on a modest thin client. Putting Professor Tux on the Nano steals shared memory the model needs and tends to push the model onto CPU. The current split — FastAPI on the thin client, Ollama on the Nano — is the right architecture; do not flip it.

Troubleshooting

1. The student page says no model is ready

  • Open /admin.
  • Check the Ollama API URL.
  • Run a model name in the model field.
  • If local Ollama is used, make sure ollama serve is running.

2. Uploaded lectures do not affect answers

  • Make sure the files actually show up in the lecture table.
  • Make sure lecture context is enabled in admin.
  • Test in Guided Learning, not Recall mode.
  • Use prompts related to the uploaded course content.

3. Recall mode refuses to answer directly

That is expected. Recall mode is intentionally constrained to hints and cues.

4. Session not found after a restart

Sessions are in memory. Restarting the backend clears them. The browser should recreate them, but old session state is not durable.

5. Cloud model output loses spaces or formatting

If you saw words being glued together when switching to cloud models, that was caused by whitespace handling in streamed chunks. That bug has already been fixed in the current code.

6. Uploads work locally but seem ignored in cloud mode

Guided mode now prefetches lecture context before generation, so uploads should still work in cloud-backed generation. If they do not, check that you are in Guided mode, lecture context is enabled, and the uploaded content matches the question.

7. Changing admin settings does not survive restart

Only some runtime settings are persisted. The Ollama URL, CTF web-search settings, and max tokens are saved to ./data/admin_runtime.json; mode/topic/course filter and lecture-context toggle are runtime-only.

8. The admin panel keeps asking me to set credentials after every restart

That is the intended behavior. Admin credentials are memory-only and are wiped on every server start by design. Sign in with the bootstrap credentials, set the new pair, and proceed. If you want different bootstrap values, change ADMIN_USERNAME / ADMIN_PASSWORD in .env before starting the server.

9. I forgot the credentials I just set in this session

Restart the FastAPI process. The in-memory username and hash are wiped, and you can sign back in with the bootstrap credentials and pick new ones.
